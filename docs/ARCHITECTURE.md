# Architecture

AI Rail has two layers: the installable public CLI and the repo-local runtime copied into each initialized project. The public command remains `rail`, and the package remains `ai-rail`.

## Code Map

| Area | Source | Responsibility |
|---|---|---|
| Public CLI wrapper | `src/ai_rail/cli.py` | Entrypoint for `rail`, short daily commands, package features, output rewriting, generated brain, exports, demo, release checks |
| Repo-local runtime | `src/ai_rail/template/.rail/rail.py` | Detailed issue workflow copied into target repos by `rail init` |
| Templates | `src/ai_rail/template/` | Files installed into a user repo, including `.rail/` contracts, default config, Makefile, and local runtime |
| Brain files | Generated `.rail/brain/*.md` | Portable project context created by `rail snapshot` and embedded by `rail handoff` |
| Exports | Root `AGENTS.md`, `CLAUDE.md`, `AIDER.md`, `.cursor/rules/ai-rail.mdc`, `.github/copilot-instructions.md` | Tool-specific instruction files generated by `rail export` |
| Safety checks | Mostly `src/ai_rail/template/.rail/rail.py`; wrapper orchestration in `src/ai_rail/cli.py` | Fresh review/check requirements, dangerous path blocking, guarded commit/ship flow |
| Tests | `tests/test_cli.py` | CLI behavior, template install behavior, safety guards, brain/handoff/export behavior, release check coverage |

## Public CLI Wrapper

Installed globally:

```text
rail CLI public layer
```

The public `rail` command lives in `src/ai_rail/cli.py`. It provides:

- short daily wrappers: `next`, `verify`, `ship`, and `resume`
- package-level commands: `about`, `demo`, `release-check`, `ci-init`, `log`, and `report`
- model-aware state around the active issue
- `.rail/brain/` generation through `snapshot`
- model-specific handoff rendering through `handoff`
- tool-specific export generation through `export`
- output rewriting for legacy local-runtime hints

Commands that are still owned by the repo-local runtime are delegated to `.rail/rail.py`. This keeps the public wrapper small while preserving compatibility with initialized repositories.

## Repo-Local Runtime

Inside each initialized repo:

```text
.rail/
  rail.py          # repo-local workflow runtime / compatibility layer
  config.json
  PROJECT.md
  AGENTS.md
  CHATGPT.md
  CLAUDE.md
  CODEX.md
  THREE_MODELS.md # installed quick reference; docs/WORKFLOWS.md is canonical
  AI_CONTRACT.md
  brain/           # generated by rail snapshot
  state/           # generated local workflow state
```

The repo-local runtime is copied from `src/ai_rail/template/.rail/rail.py` by `rail init`. It owns the detailed GitHub issue flow:

- `issue-list`, `issue-template`, `issue-create`, and `issue-comment`
- `start`, `prompt`, `review`, `checks`, and `patch`
- `commit`, `issue-close`, `pr-create`, `done`, and `sync`

The public wrapper calls this runtime with subprocess delegation for detailed workflow behavior, then adds higher-level orchestration around it.

## Templates

`src/ai_rail/template/` is source, not generated state. Edits here change what future `rail init` installs into user repositories.

Important template files:

- `src/ai_rail/template/.rail/rail.py`: copied local runtime
- `src/ai_rail/template/.rail/config.json`: default project config
- `src/ai_rail/template/.rail/AGENTS.md`, `CODEX.md`, `CHATGPT.md`, `CLAUDE.md`, `AI_CONTRACT.md`: installed AI guidance
- `src/ai_rail/template/.rail/THREE_MODELS.md`: compact installed workflow reminder
- `src/ai_rail/template/.rail/.gitignore` and `state/.gitignore`: local-state ignore rules

Do not treat `.rail/state/` files created in initialized repos as source. Only template files under `src/ai_rail/template/.rail/` should be edited as part of product changes.

## Brain Files

`rail snapshot` writes:

```text
.rail/brain/PROJECT.md
.rail/brain/CURRENT_TASK.md
.rail/brain/STATUS.md
.rail/brain/RECENT_HISTORY.md
.rail/brain/HANDOFF.md
```

The wrapper builds these from config, active issue state, git status, changed files, last checks, and local history. `rail handoff` renders the same project brain into paste-ready prompts for `generic`, `chatgpt`, `codex`, `claude`, `cursor`, or `aider`.

## Exports

`rail export` turns the project brain into tool-specific instruction files:

```text
AGENTS.md
CLAUDE.md
AIDER.md
.cursor/rules/ai-rail.mdc
.github/copilot-instructions.md
```

Exported files use managed markers:

```text
<!-- AI_RAIL_EXPORT_BEGIN -->
<!-- AI_RAIL_EXPORT_END -->
```

When markers are present, AI Rail updates only the managed block. When an existing file has no markers, export refuses to overwrite it unless `--force` is used, and `--force` first writes a `.rail.bak` backup.

## Safety Checks

Safety behavior is part of the product contract. Do not remove or weaken it during cleanup.

Key guardrails:

- `rail commit` and `rail ship` require fresh review context unless explicitly bypassed.
- `rail commit` and `rail ship` require fresh checks unless explicitly bypassed.
- dangerous/generated paths such as `.env`, keys, local databases, `node_modules/`, `dist/`, and `.rail/state/` are blocked by default.
- `rail ship` composes commit, issue close, done, and sync while preserving the underlying commit guards.
- model guards prevent accidentally using Codex prompts for patch or AI-direct tasks unless forced.

Most low-level commit/check/path enforcement lives in the repo-local runtime. The public wrapper coordinates the short commands and passes through explicit escape hatches such as `--allow-missing-checks`, `--allow-stale`, and `--force`.

## Tests

`tests/test_cli.py` is the main regression suite. It exercises:

- `rail init` template installation and stack defaults
- model guard behavior
- CI workflow generation
- history log/report behavior
- output rewriting from local-runtime hints
- safety guards around review, checks, and dangerous files
- snapshot, handoff, and export generation
- release-check readiness
- dry-run export behavior

Run the full suite after behavior changes and after broad documentation reorganizations that affect release checks:

```bash
python -m pytest -q
```

If your terminal does not default to UTF-8, set `PYTHONIOENCODING` for the test run.

macOS/Linux:

```bash
PYTHONIOENCODING=utf-8 python -m pytest -q
```

Windows PowerShell:

```powershell
$env:PYTHONIOENCODING='utf-8'; python -m pytest -q
```

## Maintainer Notes

- Keep `VERSION` in `src/ai_rail/cli.py`, `pyproject.toml`, and the template runtime aligned only during explicit version work.
- Keep the public command `rail`; do not reintroduce old aliases or rename commands in docs.
- Keep workflow docs centralized in `docs/WORKFLOWS.md`; installed `.rail/THREE_MODELS.md` is only a compact reference.
- Keep `docs/COMMANDS.md` aligned with both wrapper-owned commands and delegated runtime commands.
- Prefer narrow docs or tests for cleanup tasks. Avoid broad refactors unless the user explicitly asks for one.

## Cross-Platform Notes

- Use `pathlib.Path` for filesystem work and avoid hardcoded platform paths.
- Run subprocesses with argument lists unless intentionally executing user-configured check commands.
- `rail checks` runs configured check strings with the platform shell; keep default checks simple and documented.
- Clipboard support is best-effort: Windows `clip`, macOS `pbcopy`, and common Linux clipboard tools are attempted when available.
- The installed `Makefile` is a convenience for environments with `make`; the `rail` CLI remains the portable interface.
- Git and GitHub CLI integration depends on `git` and `gh` being installed and available on `PATH`.

## History Privacy

`.rail/state/history.jsonl` is generated locally and ignored by default.

Solo users can opt in to tracking it by editing `.rail/.gitignore`. Team repos should usually keep it local-only.

## Workflow Documentation

The three interaction models are documented in [WORKFLOWS.md](WORKFLOWS.md). The installed `.rail/THREE_MODELS.md` file is a compact in-repo reminder for initialized projects.
